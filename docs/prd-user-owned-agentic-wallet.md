# PRD：ユーザー所有エージェンティックウォレット（Option B）

> エンベデッドウォレット + Delegated Signer による自律エージェント委任モデル

**バージョン:** 1.0
**作成日:** 2026-03-02
**ステータス:** ドラフト

---

## 1. エグゼクティブサマリー

現状のシステムでは、AIエージェントが操作するサーバーウォレットはアプリが所有しており、ユーザー間でウォレットが混在する設計になっている。本PRDでは、各ユーザーが自分自身のウォレットを完全に所有しつつ、AIエージェントに委任署名（Delegated Signing）権限を付与することで、エージェントが自律的にユーザーの代わりに行動できるアーキテクチャへの移行を定義する。

**核心的な設計思想：**
```
現状：  アプリ → サーバーウォレット（アプリ所有）→ AI操作
目標：  ユーザー → エンベデッドウォレット（ユーザー所有）→ 委任 → AI操作
```

---

## 2. 問題定義

### 2.1 現状の課題

| 問題 | 影響 |
|------|------|
| ウォレットがアプリ所有 | ユーザーは秘密鍵にアクセスできず、真の所有権を持てない |
| ユーザー概念がない | `list_wallets` で全ユーザーのウォレットが混在して見える |
| 認証なし | 誰でもチャットにアクセスし、エージェント機能を使用できる |
| エクスポート不可 | ウォレットを他のアプリで使えない |

### 2.2 ユーザーが求めるもの

- 自分のウォレットアドレスを確実に所有したい
- 秘密鍵をエクスポートして MetaMask 等で使いたい
- AIエージェントに代わりに行動させつつ、最終的な制御権は自分が持ちたい
- 他のユーザーのウォレットとは完全に分離されたい

---

## 3. ソリューション概要：Option B

### 3.1 アーキテクチャ

**Privy Embedded Wallet + Delegated Signer** を組み合わせた委任モデル。

```
┌─────────────────────────────────────────────┐
│                  ユーザー                    │
│  ┌───────────────────────────────────────┐  │
│  │       エンベデッドウォレット           │  │
│  │  ・ユーザーが秘密鍵を所有              │  │
│  │  ・秘密鍵エクスポート可能              │  │
│  │  ・Privy が MPC でセキュアに管理       │  │
│  └────────────────┬──────────────────────┘  │
└───────────────────┼─────────────────────────┘
                    │ 委任（Delegated Signer 登録）
                    ▼
┌─────────────────────────────────────────────┐
│              アプリサーバー                  │
│  ┌───────────────────────────────────────┐  │
│  │       Delegated Signer                │  │
│  │  ・ユーザーの許可を受けてサーバーが   │  │
│  │    ウォレットに代理署名               │  │
│  │  ・ユーザー確認なしにトランザクション  │  │
│  │    を実行できる（委任範囲内で）       │  │
│  └────────────────┬──────────────────────┘  │
└───────────────────┼─────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│              AIエージェント                  │
│  ・ユーザーの代わりに自律的に実行            │
│  ・send_eth, sign_message, buy_product 等    │
│  ・支出制限ポリシーで安全性を保証            │
└─────────────────────────────────────────────┘
```

### 3.2 Option A との比較

| 観点 | Option A（サーバーウォレット＋ユーザーID紐付け） | Option B（エンベデッド＋委任署名）|
|------|------|------|
| 秘密鍵の所有者 | Privy（アプリ管理） | ユーザー |
| エクスポート | 不可 | 可（MetaMask等へ移行可能） |
| 実装難易度 | 低 | 中 |
| ユーザー体験 | シンプル | より透明性が高い |
| 真の所有権 | ✗ | ✓ |
| 自律エージェント対応 | ✓ | ✓（委任署名で実現） |

## 4. ユーザーストーリー

### 4.1 サインアップ・ログイン

```
ユーザーとして、
  メールアドレス / ソーシャルログイン / パスフレーズで
  エンベデッドウォレットを作成しサービスにログインしたい。
これにより、秘密鍵の管理を気にせず Web3 ウォレットを持てる。
```

### 4.2 エージェントへの委任

```
ユーザーとして、
  AIエージェントに私のウォレットを代わりに操作する権限を与えたい。
  ただし、毎回の確認は不要で自律的に動いてほしい。
これにより、私がいない間もエージェントが代わりに行動できる。
```

### 4.3 ウォレット管理

```
ユーザーとして、
  自分のウォレットアドレス・残高・トランザクション履歴を確認したい。
  また、秘密鍵をエクスポートして他のウォレットアプリで使いたい。
これにより、サービスに依存せずウォレットを持ち続けられる。
```

### 4.4 エージェント操作の透明性

```
ユーザーとして、
  エージェントが私の代わりに実行したトランザクションを確認したい。
  また、必要に応じて委任権限を取り消したい。
これにより、エージェントの行動を把握・制御できる。
```

## 5. 機能要件

### 5.1 認証・ウォレット作成

- **FR-01**: ユーザーはメール OTP でサインアップできる
- **FR-02**: ユーザーはソーシャルログイン（Google、Apple）でサインアップできる
- **FR-03**: サインアップ時に自動でエンベデッドウォレットが生成される
- **FR-04**: ウォレット生成は Privy の MPC（マルチパーティ計算）で保護される
- **FR-05**: ユーザーはリカバリーフレーズ経由で秘密鍵をエクスポートできる

### 5.2 Delegated Signer 設定

- **FR-06**: ログイン後、ユーザーはエージェントへの委任に同意するフローを経る
- **FR-07**: 同意後、サーバーの Delegated Signer がユーザーのウォレットに登録される
- **FR-08**: 委任はいつでもユーザーが取り消せる
- **FR-09**: 委任範囲は支出制限ポリシー（最大 0.001 ETH/tx）で制約される

### 5.3 エージェント機能（ユーザースコープ）

- **FR-10**: `list_wallets` はログインユーザーのウォレットのみ返す
- **FR-11**: `send_eth`, `sign_message`, `buy_product` はユーザーのエンベデッドウォレットで実行される
- **FR-12**: エージェントはユーザー確認なしにトランザクションを実行できる（委任範囲内）
- **FR-13**: 全トランザクションはユーザーIDと紐付けてログに記録される

### 5.4 UI

- **FR-14**: 未ログイン時はログイン画面を表示し、チャットUIへアクセスできない
- **FR-15**: ログイン後、ヘッダーにウォレットアドレス（省略形）とログアウトボタンを表示する
- **FR-16**: 初回ログイン時、委任同意モーダルを表示する

---

## 6. 非機能要件

- **NFR-01**: 秘密鍵はサーバーに平文で保存されない（Privy MPC が管理）
- **NFR-02**: 支出制限ポリシーはサーバー側で強制される（クライアント改ざん不可）
- **NFR-03**: ユーザーセッションは Privy の JWT で検証される
- **NFR-04**: エージェントAPIは認証済みユーザーのリクエストのみ受け付ける

---

## 7. スコープ外（今回対象外）

- マルチチェーン対応（Ethereum Mainnet、Polygon 等）——Base Sepolia テストネットに限定
- ウォレット間の送金 UI（チャット経由のみ）
- エージェント実行履歴の専用ダッシュボード
- スマートコントラクトウォレット（ERC-4337）
- USDC 等ステーブルコイン対応

## 8. 技術アーキテクチャ

### 8.1 使用する Privy SDK

| SDK | 役割 |
|-----|------|
| `@privy-io/react-auth` | ユーザー認証、エンベデッドウォレット生成、委任同意UI |
| `@privy-io/node` | Delegated Signer としてサーバー側でトランザクション実行 |

### 8.2 Delegated Signer の仕組み

Privy のエンベデッドウォレットは、**Delegated Signer** という仕組みをサポートする。

1. ユーザーがウォレットを作成
2. アプリのサーバー公開鍵を Delegated Signer として登録（ユーザー同意が必要）
3. 以降、サーバーはユーザーのウォレットアドレスから署名・トランザクションを実行できる
4. ユーザーは Privy ダッシュボードから Delegated Signer を確認・削除できる

```typescript
// クライアント側：委任同意
const { delegateWallet } = usePrivy();
await delegateWallet({ address: embeddedWalletAddress, chainType: "ethereum" });

// サーバー側：Delegated Signer として実行
await privy.walletApi.ethereum().sendTransaction(walletAddress, {
  caip2: "eip155:84532",
  params: { transaction: { to, value, chain_id: 84532 } }
});
```

### 8.3 データフロー

```
1. ユーザーが Privy でログイン
   → userId, embeddedWalletAddress を取得

2. /api/agent にリクエスト送信
   → Authorization: Bearer <privy_access_token>

3. サーバーが Privy JWT を検証
   → userId を確認

4. Claude がツールを呼び出し
   → ツールはユーザーの embeddedWalletAddress を使って実行

5. Delegated Signer としてトランザクション実行
   → ユーザーのウォレットから送信
```

### 8.4 変更対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `package.json` | `@privy-io/react-auth` を追加 |
| `.env.example` | `NEXT_PUBLIC_PRIVY_APP_ID` を追加 |
| `app/layout.tsx` | `PrivyProvider` でラップ、embeddedWallets 設定 |
| `app/page.tsx` | 認証ゲート、委任同意フロー、ヘッダーUI を追加 |
| `app/api/agent/route.ts` | Privy JWT 検証、ユーザースコープでツール実行 |
| `lib/tools.ts` | ツールがサーバーウォレット ID の代わりに `walletAddress` を受け取るよう変更 |

## 9. ユーザーフロー

### 9.1 初回サインアップ

```
[ランディング画面]
    → [Privy ログインモーダル]（メール/Google/Apple）
    → [エンベデッドウォレット自動生成]
    → [委任同意モーダル]
        「AIエージェントがあなたのウォレットを代わりに操作します。
         支出上限: 0.001 ETH/トランザクション」
        [同意する] → Delegated Signer 登録
    → [チャットUI] ← ウォレットアドレス表示
```

### 9.2 リターンユーザー

```
[ランディング画面]
    → [Privy ログインモーダル]（以前の認証情報）
    → [チャットUI]（委任済みのため同意モーダルはスキップ）
```

### 9.3 エージェント操作

```
ユーザー: 「残高を確認して」
    → /api/agent POST（Bearer token 付き）
    → JWT 検証 → userId 確認
    → Claude: get_balance ツール呼び出し
    → Delegated Signer でユーザーのウォレット残高取得
    → レスポンスをチャットに表示
```

## 11. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| Delegated Signer の過剰権限 | 高 | 支出ポリシーで 0.001 ETH/tx を強制 |
| JWT 検証漏れ | 高 | 全APIルートでサーバー側検証を必須化 |
| ユーザーが委任を理解しない | 中 | 同意モーダルで明確な説明を表示 |
| テストネット ETH 不足 | 低 | フォーセットリンクをチャットUIに表示 |
